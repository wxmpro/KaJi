## 1. 已修 P0 项（6/7）

### P0-1: `CardService.generateNewCard` 主线程同步读 SQLite

**根因**:
- 位置: `Services/CardService.swift:39-43`（修改前）
- 每次新建卡（⌘N / 侧栏"新建卡片"按钮 / 启动 init）在 `@MainActor` 上**同步**读 SQLite 全表 id (`dbWriter.read`)
- 1k 卡 ≈ 5-15ms 阻塞主线程；10k 卡 ≈ 50-150ms 阻塞

**修复**:
- `CardService.generateNewCard` 改 `async throws`，`AppDatabase.shared.allIDs()` 包进 `Task.detached(priority: .userInitiated)`
- `EditorState.startNewCard` 和 `init` 改 `Task { @MainActor in try await ... }` 包裹
- 文件: `Services/CardService.swift`、`App/EditorState.swift`

**关联 bug 修复（横向扫描发现）**:
- ⌘N 多窗口 bug（v1.2.5 既有）
- 旧版 `startNewCard` 同步阻塞主线程，Button handler 跑完才返回，SwiftUI 不触发 WindowGroup 默认 New
- P0-1 改成 async 后 handler 立刻返回，SwiftUI fallback 到 WindowGroup 默认 New → 每次 ⌘N 开新窗口
- 修复: `KaJiApp` 的 `CommandGroup(after: .newItem)` 改 `CommandGroup(replacing: .newItem)`，替换 WindowGroup 默认"新建窗口"项
- 文件: `App/KaJiApp.swift`

**性能影响**:
- 1k 卡: ~10ms 阻塞 → **0ms**
- 10k 卡: ~100ms 阻塞 → **0ms**
- 启动期 init 不再阻塞

**UI 影响**: 0（除修 ⌘N 多窗口 bug）

### P0-2: `CardFileIO.parseISO` 每次解析都 new ISO8601DateFormatter

**根因**:
- 位置: `Database/CardFileIO.swift:216-222`（修改前）
- reconcile 期间每个 .md 解析 frontmatter 会调 parseISO 4-5 次
- 每次都 `let f = ISO8601DateFormatter()`，创建成本 ~50-100µs
- 1k .md 库启动期累计 ~400ms 浪费

**修复**:
- 复用上面的 `isoFormatter` 静态缓存
- 第一次失败（字符串无 fractional seconds）才 new fallback
- 文件: `Database/CardFileIO.swift`

**性能影响**:
- 100 .md: ~40ms → **0ms**
- 1k .md: ~400ms → **0ms**
- 10k .md: ~4s → **0ms**（实际节省更显著）

**UI 影响**: 0

### P0-3: `CardRepository.reconcile` missingInMD 循环 N+1 查询

**根因**:
- 位置: `Database/CardRepository.swift:240-250`（修改前）
- 缺 N 张 .md → 调 N 次 `self.card(id:)` → N 次独立 `dbWriter.read` + 每次走 hydrate
- 数据库 IO O(N) 次 + 锁等待 + 串行

**修复**:
- 一次 `self.allCards(includeDeleted: true)` 拉全量 + `Dictionary(uniqueKeysWithValues:)` 建索引
- 数据库 IO 从 O(N) 次降到 O(1) 次
- 文件: `Database/CardRepository.swift`

**性能影响**:
- 缺 10 张 .md: ~50-100ms → ~5-10ms
- 缺 100 张 .md: ~500-1000ms → ~10-20ms

**UI 影响**: 0（reconcile 在 Task.detached 后台跑，UI 不阻塞）

### P0-4: `CardRepository.save` 与 `EditorState.persistCurrentCard` 双重截断

**根因**:
- 位置: `App/EditorState.swift:144-147` + `Database/CardRepository.swift:55-57`（修改前）
- 主线程 `persistCurrentCard` 先 `isOverLimit + truncate` 一次（为了 `currentCard = c` 同步 UI）
- 后台 `CardRepository.save` 内部又 `isOverLimit + truncate` 一次
- 每次保存 O(N) 字符统计 × 2

**修复**:
- 删掉 `CardRepository.save` 内部截断（信任 caller）
- 注释明确: **截断是 caller 的责任**（`EditorState.persistCurrentCard` 已在主线程截断）
- 文件: `Database/CardRepository.swift`

**性能影响**:
- 每次保存省一次 O(N) 字符统计（~10-100µs）
- 主要价值是**责任清晰**（caller / callee 不再重复）

**UI 影响**: 0

### P0-5: `StatsState.update` + `ListState.refreshFilteredCards` 触发全订阅者级联刷新

**根因**:
- 位置: `App/StatsState.swift:38-46` + `App/ListState.swift:55-57`（修改前）
- 每次 `update(with:)` 把 cachedCards / cachedTypeCounts / cachedTagCounts 全部赋值
- SwiftUI `ObservableObject` 的 `@Published` **不做 diff**，值相等也触发 objectWillChange
- 编辑标题（只改 title）时，typeCounts / tagCounts / cachedCards 实际未变，但触发 SidebarView / CardListView / ListFilterTitle 全部 body 重算
- 同样 `refreshFilteredCards` 每次都重发

**修复**:
- `StatsState.update` 三个 @Published 赋值前用 `if newValue != oldValue` 守门
- `ListState.refreshFilteredCards` 同样守门
- 注: `[(String, Int)]` 元组数组在 Swift 6 下不能直接走 Equatable 协议，用 zip + `==` 元素比较做 O(N) 短路
- 文件: `App/StatsState.swift`、`App/ListState.swift`

**性能影响**:
- 编辑标题（typeCounts / tagCounts 不变）: 3 个 @Published 全发 → **0 个发**
- 编辑字段值: 同上
- 切换卡片类型: 只变了的发，没变的短路
- 软删除一张卡: 同上

**UI 影响**: 0

### P0-7: `KaJiApp` `NSWindow.didBecomeKeyNotification` 重复订阅

**根因**:
- 位置: `App/KaJiApp.swift:107-145`（修改前）
- `init` 里 addObserver 订阅 `NSWindow.didBecomeKeyNotification`
- `applicationDidFinishLaunching` 已调 `configureWindows()` 遍历所有窗口配置

- 重复订阅 + 每次遍历 + 每次 setter 调用

**修复**:

- 只保留 `applicationDidFinishLaunching` 里的 `configureWindows()`
- 文件: `App/KaJiApp.swift`

**性能影响**:

- 单窗口应用几乎不触发；多窗口切换时改善

**UI 影响**: 0

### P1-2: `CardService.refreshStats` typeDict 计算 O(12N)

**位置**: `Services/CardService.swift:80-83`

```swift
for type in CardType.allCases {  // 12 个
    typeDict[type] = cards.filter { $0.cardType == type && $0.deletedAt == nil }.count
```

```swift
    typeDict[card.cardType, default: 0] += 1
}
```

**风险**: 零

**预期收益**: refreshStats 时间从 O(12N) 降到 O(N);N=10k 时从 ~5ms 降到 ~0.5ms

**建议**: 未修,等用户决定（小且明显）

## 6. 用户操作暴露的非 P0 bug

### Bug-A: "返回后该卡片下面有灰色底"（已知，未修）

**用户报告**: 在列表点一张卡 → 不修改 → 返回 → 那张卡片下面有灰色底

**根因**:
- `CardListView` 的 `List(selection:)` 绑定 `editorState.currentCard?.id`
- 用户点列表卡 → `currentCard = card` → 进入 editor
- 用户返回 → `rightPaneMode = .list` → List 重新出现
- 此时 `currentCard` 仍指向用户点的那张卡，List 重新出现时 SwiftUI 看到 selection 指向的 id 在 cards 列表里 → 系统高亮该行（灰色底）
- 这是 v1.2.5 原状行为

**用户用 git stash 撤回我所有改动，亲眼在 v1.2.5 原状下确认此 bug 存在**

**为什么我两次尝试修复都引入了新问题**:
- 第一次: backButton 加 `flushSave()` → 即使无修改也触发 `currentCard = c` + `statsState.update` → List 重建 + selection 漂移到第一行
- 第二次: backButton 加 `if hasPendingSave { flushSave() }` + openCard 同样条件 flush → 仍有副作用
- **真正根因**是 `persistCurrentCard` 内部 `currentCard = c` 无条件触发 @Published（**P0-5 守门仅针对 StatsState / ListState，未针对 EditorState.currentCard**）
- 这需要更深的状态拆分（P0-6 范畴），用户决定"不做"

**当前状态**: 未修。用户接受 v1.2.5 原状行为。P0-5 守门**间接降低**了相关问题出现概率（refreshFilteredCards 无变化时不发，List 不重建，selection 不会漂移），但"返回后该卡高亮"本身是 v1.2.5 行为，不做修改。
