## 0. 一句话总评

**v1.1.0 整体架构清晰、双存储策略（Markdown + SQLite）合理、UI 完全走原生 SwiftUI 组件、没有任何硬编码颜色。** 主要问题集中在**单文件膨胀**（MainView 891 行 / 15 个 nested private struct）、**AppState 过度集中**（30+ `@Published`）、**FTS5 索引与内存缓存两套搜索结果存在不一致风险**、**GRDB 同步写盘在 @MainActor 上**四个方向。

按"四要素"框架回答你的检查清单：

- **优化点**：MainView 拆分、AppState 拆职责、搜索统一入口、GRDB 写盘线程化
- **目的**：消除单文件/单类的认知负担，规避"卡顿 / 搜索结果不一致 / 数据丢失"三类隐患
- **预期成效**：MainView 行数从 891 → ≤ 300；AppState 从 423 → ≤ 200；搜索响应 P99 ≤ 50ms；零崩溃
- **边界**：仅做**结构性整理与已知 BUG 防护**，**不**重构 UI 视觉、**不**改协议（DB 表结构、Card 字段、GRDB 记录类型）、**不**改 PRD（11 种卡型、3500 字上限、5×10 标签）

## 3. 必须修的隐患（Critical / High）

### 🚨 H-1：FTS5 搜索与内存搜索结果不一致

| 项 | 值 |
|---|---|
| 严重度 | **High** |
| 位置 | `App/AppState.swift:262-270`、`398-422`；`Database/CardRepository.swift:174-190` |
| 现状 | `AppState.search(_:)` 走 `cachedCards.filter`（`localizedCaseInsensitiveContains`），`AppState.filteredCards()` 的 `.search` 分支也走同样内存路径；`CardRepository.search(_:)` 走 FTS5（trigram）。**两条路径并存**，搜索框走前者，外部调用方走后者 |
| 风险 | ① 中文 2 字短词 R-04 已知召回不全，UI 上看到 0 结果时用户无法判断"是没找到"还是"FTS 不会召回" ② 后续任何路径切换（按 Enter 走 `showList(.search)` 还是 `repository.search`）会再次切换行为 ③ 单元测试不可行（两条路径要测两遍） |
| 建议 | **二选一收敛**：<br>**A. 推荐**：删除 `CardRepository.search` 的 FTS5 实现，`AppState.search` + `filteredCards` 统一走 `cachedCards.filter`，并缓存搜索结果。代价：放弃 FTS5 性能。<br>**B. 兜底**：FTS5 升级 trigram 3 字符起步（已部分做），并把 `AppState.search` 改为 `@MainActor` 异步 + Task 包裹，让两侧行为一致。<br>**C. 现状延续**：写明注释 + 写一份对照测试，但**禁止出现第三条路径**。 |
| 验证 | 输入"心流" → 比较 `AppState.search` vs `repository.search` 结果数量；输入"Deep" → 同上。两者**应严格一致**。 |

### 🚨 H-2：GRDB 同步写盘在 `@MainActor` 上执行

| 项 | 值 |
|---|---|
| 严重度 | **High** |
| 位置 | `App/AppState.swift:121-138`（init）、`203-227`（`persistCurrentCard`）、`283-300`（`rebuildStats`） |
| 现状 | `AppState` 是 `@MainActor`，`init` 里同步调用 `repository.bootstrap()` → `db.purgeOldTrash()` 写盘；`saveImmediately()` 800ms debounce 后**在 main thread 调 `repository.update` 同步写盘 + 重算 `rebuildStats()`** |
| 风险 | ① 大数据量（3500 字上限 × 几千张卡）下 `update` 写盘 + 重建全量缓存可能造成**主线程可感卡顿**（CLAUDE.md 第 1 节「回归」明确指出卡顿是已确认优化项，不能破坏） ② init 阶段在 main thread 触发 GRDB 写 → 万一数据库文件锁竞争（用户同时打开两个 app 实例），主线程会卡 1–3s |
| 建议 | `repository.update` 改为**显式后台队列**（`DispatchQueue.global(qos: .utility)` 或 GRDB 自带 `dbWriter.write { ... }` 内部已异步）后，**让 main thread 只更新 `@Published` 状态**。`rebuildStats` 拆为 `loadFromDB`（后台）+ `applyToCache`（main）。 |
| 验证 | 用 Instruments → Time Profiler 录一次"输入一段 2000 字内容"的操作，确认 main thread 不再出现 > 50ms 的同步写盘调用栈。 |

## 5. 低优先级建议（Low）

### L-1：硬编码颜色值（CardType）

`Models/CardType.swift:79-94` 用 11 个 `Color(red:green:blue:)` 硬编码，而不是 `Color(nsColor: .systemRed)` 之类的系统色。CLAUDE.md 明确说"颜色 → `NSColor` 系统语义色"。**但**这是 macOS 调色板的"设计 token"色，不是 UI 控件色（hover / separator / controlBackground），设计意图上**可以接受**。但建议在注释中说明"这些是品牌色，不跟随系统"。

### L-2：`Card.cardType` 每次访问反查 `CardType(rawValue:)`

`Models/Card.swift:31` `var cardType: CardType { CardType(rawValue: type) ?? .free }` — 每次访问都解析。在热路径（list 渲染）会有 N×M 次调用。**目前数据量小（< 1k 卡）无影响**，未来 ≥ 10k 卡时可考虑缓存到 `Card` 字段。

### L-3：`ExportService` NSSavePanel / NSOpenPanel 阻塞主线程

`Utilities/ExportService.swift:21, 39` 用了 `panel.runModal()` 阻塞主线程。macOS 用户可接受（导出是低频操作），但导出多卡时循环 N 次写文件会**在主线程**串行写。**可接受**。

### L-4：`ContentLimit.truncate` 在 `dropLast` 时按字符截断

`Utilities/ContentLimit.swift:46` `String(f.fieldValue.dropLast(min(over, f.fieldValue.count)))` 是 Swift String 字符级截断。**Swift 的 `dropLast` 是 Unicode 标量级**，对绝大多数中文/emoji 表现正确（emoji 是单个标量簇的多个标量时可能截在簇中间）。**可接受**，但建议加注释说明。

### L-5：`fatalError("No main screen")` 在生产代码

`App/KaJiApp.swift:44-46` `guard let screen = NSScreen.main else { fatalError(...) }`。现代 macOS 基本不可能没主屏。**可接受**（启动期直接崩比静默失败更好定位）。

### L-6：`ContentView.swift` 是空 wrapper

`App/ContentView.swift` 整个文件只 20 行，且注释说"被 KaJiApp.swift 里 AppDelegate 直接宿主 MainView"。`#Preview { ContentView() }` 也没人在用。**可删除**。

### L-7：`Models/Tag.swift` 的 `id: Int`（非 `Int64?`）

GRDB 的 `TagRecord.id` 是 `Int64?`（自增主键），但 `Tag.id` 是 `Int`。类型不匹配 → 即使引用也无法直接转换。**印证 M-4 的删除建议**。

### L-8：`KaJiApp.Settings { SettingsView() }` 多余 Scene

`App/KaJiApp.swift:19-23` `Settings { SettingsView() }` 是 SwiftUI 标准的"⌘, 打开设置"入口，但**主 AppScene 不存在**（AppDelegate 自己创建窗口），`Settings` 单独存在 → 在 macOS 15 上点击菜单栏的"设置"会启动第二个无主窗口的进程。**可接受**（macOS 会处理），但应确认是否预期行为。
