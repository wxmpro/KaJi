# KaJi 设计宣言（Design Manifesto）

> 版本：v1.0 · 日期：2026-06-20 · 状态：初稿，待修订

这不是市场文案，也不是产品白皮书。这是一份给**用户、潜在贡献者、以及未来的自己**的承诺清单——说清楚 KaJi 是什么、不是什么、为什么这样做。

---

## 一、我们为什么做 KaJi

过去十年，记笔记的工具越来越多，但**真正改变思考方式**的很少。

主流笔记工具把人当成"容器"——输入框越来越宽、AI 自动补全越来越长、模板越来越花哨——结果是你的笔记越多，**思考反而越浅**。

KaJi 的判断是：

> **笔记的本质不是记录，是思维训练。工具的责任不是替你写，是逼你想清楚。**

基于这个判断，我们做了三个不可妥协的决定：**本地优先**、**卡片原子化**、**macOS 原生**。

---

## 二、12 张卡片 = 12 种思维训练

每种卡片类型不是"字段集合"，是**特定的认知动作**：

| 卡片 | 字段结构 | 训练什么 |
|---|---|---|
| **术语卡**（teal） | 定义 / 解释 / 例子 / 参考 | 精确定义一个概念 |
| **反常识卡**（red） | 常识 / 反常识 / 例子 / 参考 | 识别并挑战你的默认假设 |
| **新知卡**（yellow） | 已知 / 新知 / 例子 / 参考 | 把新信息与你已有的知识区分 |
| **人物卡**（blue） | 简介 / 参考 | 一个人值不值得被记住 |
| **金句卡**（orange） | 原句 / 评论 / 参考 | 区分"好听的话"与"经得起推敲的话" |
| **新词卡**（mint） | 原句 / 造句 / 参考 | 把别人的词变成自己的 |
| **行动卡**（green） | 内容 / 行动 / 参考 | 区分"想做的事"与"真正会做" |
| **事件卡**（indigo） | 时间 / 地点 / 参与者 / 经过 / 理解 / 参考 | 还原并解释一件事 |
| **图示卡** | 说明 / 参考 | 用图替代文字 |
| **索引卡**（gray） | 引用 / 参考 | 把已有卡片织成网络 |
| **综述卡**（purple） | 论点 / 参考 | 把零散想法提炼为论点 |
| **自由卡**（gray） | 内容 / 参考 | 当上面 11 种都不合适时 |

**为什么必须有 12 种，而不是 1 个空白输入框？**

因为**空白是思考的天敌**。当你看到"反常识卡 · 常识"这个字段，你的脑子会自动开始寻找——这个看似常识的事，是不是错的？

这是 KaJi 的核心方法论：**结构是脚手架，不是牢笼**。

## 三、三个不可妥协的原则

### 原则 1：本地优先（Local-First）

**你的数据是你的，不是借给你的。**

每张卡同时存储为两份：
- **SQLite**（WAL，强一致锚点，所有搜索与统计的来源）
- **`.md` 文件**（Markdown + YAML frontmatter，人可读、git 可同步、永久可访问）

`mdVersion` 字段双向对账，确保两份数据不会漂移——启动时 reconcile 修复漂移、写入时严格排序避免脏读。

**我们不会做的事**：
- ❌ 强制云同步
- ❌ "为了体验"上传你的卡到我们服务器
- ❌ 把 `.md` 文件藏起来不让你看
- ❌ 锁定格式（你能用任意文本编辑器打开、修改你的卡）

### 原则 2：macOS 原生（macOS-Native）

**不妥协的原因**：网页和 Electron 工具做不好 Mac 体验。

KaJi 深度使用 macOS 26 新 API：
- `@Observable` 状态管理（细粒度订阅，不重渲染）
- `.searchable(placement: .toolbar)` + `NSSearchField` 原生失焦行为
- `NavigationSplitView` 系统侧栏
- Swift 6 actor 隔离（`MarkdownWriteQueue` 串行写盘）
- `.windowToolbarStyle(.unifiedCompact)` 统一工具栏

**我们不会做的事**：
- ❌ 跨平台（Windows、Linux、iOS）——至少在 v2.0 之前不会做
- ❌ 用 Web 技术栈包装
- ❌ 牺牲 macOS 体验换"全平台"

### 原则 3：个人开发者的纪律

**没有大厂资源，所以每行代码都要算账。**

v1.6.11，47 个 release，平均每 4 天一个版本。每一次发布都是：
- 版本号严格递增（semver）
- commit message 列出本版本核心改动
- git tag + 不覆盖
- 推送前 build + 用户实测验证

**我们不会做的事**：
- ❌ "先推上去再修"
- ❌ "这次先不管测试"
- ❌ "build 通过就算交付"
- ❌ `git reset --hard` 销毁未推送的工作

## 四、设计红线（绝不让步的 UI 规则）

这些不是"风格选择"，是**产品语义**——改了就破坏用户的肌肉记忆。

1. **traffic-lights 必须在侧栏内**——窗口控制属于内容区，不属于装饰区
2. **搜索按钮在右上角**——与 traffic-lights 同一水平线，紧贴窗口右边缘
3. **搜索从按钮左侧向左展开**——按钮位置不动，搜索框从右往左生长
4. **点击外部关闭用 NSSearchField 原生失焦**——不用覆盖层、不用 simultaneousGesture
5. **两种模式（亮 / 暗）必须视觉、颜色、动效统一**——不允许"白天好看晚上别扭"
6. **侧栏切换按钮必须保留系统自动生成的那一个**——不自己再加

每条红线的存在都是因为我们试过别的写法，**别的写法让用户痛苦**。

---

## 五、我们不做什么（Anti-Features）

明确说出"不做"比"做"更难，但同样重要：

- ❌ **不做 AI 自动写卡**——会让人变成 AI 的校对员，违背"逼你想清楚"的承诺
- ❌ **不做云协作**——会破坏本地优先承诺
- ❌ **不做"社交分享卡"**——你的卡片是你的思考，不是内容农场
- ❌ **不做无限模板市场**——12 种卡片类型够用，多了反而稀释每种类型的训练强度
- ❌ **不做跨平台 iOS / iPadOS app**（v2.0 之后再议）——先把 Mac 做透
- ❌ **不做"打开即弹窗营销"**——你已经买了，为什么还要被卖

## 六、接下来的路（Roadmap）

| 版本 | 时间窗 | 核心内容 |
|---|---|---|
| **v1.7** | 2026 Q3 | 公开 `.md` 卡片样例库 · 单元测试覆盖核心逻辑（双写对账 / 状态机 / EAV） |
| **v1.8** | 2026 Q4 | 用户反馈通道（GitHub Issues + 应用内直链） · Sparkle 自动更新 |
| **v2.0** | 2027 H1 | 双链 / 文件夹（v1.0 注释里就承诺过、至今未做） · iCloud Drive 同步 |

每个版本发布前，本宣言会被重新审视——如果某个原则过时了，**改原则，不要悄悄破坏**。

如果你读到这里，谢谢你。

KaJi 不试图取悦所有人。我们取悦的是这样一类人：

> **愿意为"想清楚一件事"付出 5 分钟，而不是让 AI 用 5 秒钟生成一段看起来像思考的文字。**

如果你也是这样的人，欢迎。
